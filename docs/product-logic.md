# LLM 模型 API 管理器：产品逻辑文档

## 1. 产品目标

`LLM 模型 API 管理器` 的目标是帮助用户以可视化方式统一管理多家模型供应商与模型配置，并完成以下闭环：

1. 配置供应商与模型
2. 验证模型可用性
3. 导出到目标工具使用
4. 通过 JSON 做备份、迁移与团队共享

产品定位偏向“本地配置中控台”，强调低门槛、可迁移、可导出，而不是在线托管服务。

## 2. 典型用户与场景

### 2.1 目标用户

- 个人开发者：同时使用 OpenAI 兼容和 Anthropic 兼容 API
- AI 应用重度用户：需要在多个客户端之间同步模型配置
- 团队成员：希望共享一套标准 Provider/Model 模板

### 2.2 典型场景

- 新增供应商并添加多个模型做可用性验证
- 批量选择模型导出到指定 AI 客户端
- 在新电脑上导入旧电脑导出的 JSON 快速恢复配置
- 合并同事发来的配置而不覆盖本地已有模型

## 3. 核心对象模型

### 3.1 Provider（供应商）

核心字段：

- `id`：唯一标识（合并和去重基准）
- `name`：显示名称
- `apiType`：接口类型（`openai` / `anthropic`）
- `url` + `endpoint`：请求地址
- `apiKey`：可选密钥
- `website`：可选官网
- `models`：模型列表

### 3.2 Model（模型）

核心字段：

- `id`：模型唯一标识（同 Provider 内合并基准）
- `name`：显示名称
- `contextWindow` / `maxTokens`：可选参数
- `reasoningMode` / `inputTypes`：能力相关参数
- `status` / `lastCheck` / `lastMessage`：运行态检查结果

## 4. 页面与交互逻辑

## 4.1 主页面结构

- 顶部：全选、已选计数、批量检查、批量导出
- 中部：Provider 列表与每个 Provider 下的 Model 列表
- 底部左侧：`+ 添加供应商`
- 底部右侧：`导入 JSON 配置`、`导出 JSON 配置`

### 4.2 Provider/Model 管理

- Provider 支持新增、编辑、删除、复制、拖拽排序
- Model 支持新增、编辑、删除、拖拽排序
- 勾选模型用于批量检查和批量导出

### 4.3 状态反馈

- 模型状态：未检查 / 成功 / 失败
- 失败时支持查看错误详情
- 导出弹窗支持内容预览及后续动作（复制/执行）

## 5. 检查与探测逻辑

### 5.1 模型可用性检查

- 根据 Provider 的 `apiType` 组装请求头与请求体
- 对单模型或选中模型发起检测
- 把结果写回模型运行态字段并刷新界面

### 5.2 模型能力探测

- 尝试从常见模型元数据端点获取能力信息
- 提取 `contextWindow` 与 `maxTokens`
- 根据模型命名规则推断 `reasoningMode` 与 `inputTypes`

## 6. 导出逻辑

### 6.1 导出入口

- 用户勾选模型后点击 `导出选中模型`
- 在导出弹窗中选择导出器格式

### 6.2 导出结果类型

导出器可返回多种类型内容（如命令、链接、Markdown、JSON 等），预览层按类型渲染：

- 代码类：语法高亮
- Markdown：按 Markdown 渲染
- 可执行类型：提供“运行”动作

### 6.3 运行动作原则

- Deep Link：调起外部应用
- Command/Env：执行前二次确认（提醒风险）
- JavaScript：生成临时 HTML 后在浏览器执行

## 7. JSON 导入/导出逻辑

### 7.1 导出 JSON

- 基于当前内存配置生成标准 JSON
- 包含 `version`、`exportedAt`、`providers`
- 下载为本地文件（便于备份/分享）

### 7.2 导入 JSON

支持两种输入结构：

- 直接数组：`[...]`
- 包装对象：`{ "providers": [...] }`

导入后统一经过标准化流程，再更新界面与本地存储。

### 7.3 冲突处理（重点）

当“当前已有配置”且“导入文件也有配置”时，弹窗提供三种选择：

- `合并导入`
- `覆盖导入`
- `取消`

#### 合并导入规则

1. 按 `provider.id` 合并 Provider
2. 若 Provider ID 相同：
   - Provider 级字段以导入数据为准
   - 同一 Provider 内按 `model.id` 合并
3. 若 Model ID 相同：以导入模型为准
4. 不冲突项保留，最终取并集

#### 覆盖导入规则

- 直接以导入数据替换当前全部配置

## 8. 数据存储与安全边界

### 8.1 本地存储

- 使用 Electron 主进程配置文件存储（`app.getPath('userData')`）
- 默认文件名：`configs.json`

### 8.2 安全边界

- API Key 仅在本地用于请求与导出内容生成
- 不主动上传到项目外部服务
- 命令执行类导出在触发前给出风险确认

## 9. 错误处理策略

- 文件导入异常：显示明确错误提示（格式不合法、字段缺失等）
- 检查失败：将错误回写到模型状态并可查看详情
- 外部调起失败：提示用户检查协议或系统环境

## 10. 后续演进建议

- 合并策略可配置化（例如 Provider 字段按“仅导入非空覆盖”）
- 增加导入预览（展示将新增/覆盖的对象）
- 增加 JSON Schema 校验与版本迁移器
- 增加“仅导入选中 Provider/Model”能力
