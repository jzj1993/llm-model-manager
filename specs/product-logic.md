# LLM 模型 API 管理器：产品逻辑文档

本文档只描述**整体功能与闭环**，不展开字段约定、弹窗交互、预设合并规则等细节；后者见专项说明（如供应商与模型相关文档）。

---

## 1. 产品目标与定位

**LLM 模型 API 管理器**帮助用户在本地以可视化方式集中管理多家模型供应商与模型，并完成典型闭环：

1. 维护供应商与模型配置  
2. 验证模型是否可用  
3. 将选中模型导出到常用 AI 客户端或命令行/环境配置  
4. 通过 JSON 做备份与迁移  

定位是**本地配置中控台**：低门槛、可迁移、可导出；不提供在线托管或多端同步服务。

---

## 2. 典型用户与场景

**用户画像**：个人开发者、多客户端用户、需要在团队间传递标准配置的成员。

**常见用法**：为多个供应商配置 Base URL、Endpoint 与密钥；勾选模型做批量测试或导出；换机器时导出/导入 JSON；从供应商侧拉取模型列表后挑选加入本地列表。

---

## 3. 核心概念（对象模型）

- **Provider（供应商）**：表示一家 API 端点，包含接口类型（OpenAI 兼容或 Anthropic 兼容）、连接信息（如 Base URL、Endpoint）、可选密钥与官网，以及其下的模型列表。  
- **Model（模型）**：隶属于某一供应商，包含模型标识、展示名、可选的能力/参数（集中在 `params` 中），以及测试产生的运行态（是否成功、时间、错误信息等）。  
- **导出条目**：一次导出可产生多条内容（标题、正文、类型），供预览、复制或按类型触发系统能力（如打开链接、在终端执行等）。

具体字段与类型定义以代码中的共享类型为准；不在此罗列。

---

## 4. 整体能力概览

### 4.1 配置与列表

用户可管理多条供应商及其下的多条模型：增删改、拖拽排序、按供应商折叠/展开、全选或按供应商勾选模型。列表与弹窗中的**预设与补全**（供应商预设、模型预设、URL/Endpoint 建议等）用于降低录入成本；行为细节见专项文档。

### 4.2 连通性检查

按供应商的接口类型与连接信息，通过主进程代理发起 HTTP 请求，对单个模型或勾选的一批模型做**可用性测试**，并把结果写回界面。

### 4.3 从远端获取模型列表（可选）

在已配置供应商的前提下，可请求常见的「模型列表」HTTP 接口，在弹窗中展示返回结果；用户勾选后加入本地模型列表。返回数据会尽量映射为模型标识、名称与部分能力字段，并可能辅以命名规则推断；具体规则见实现代码与专项说明。

### 4.4 导出

用户勾选模型后选择**导出器**（面向不同工具：如环境变量、命令行、Deep Link、说明文档等），生成可预览的内容，支持复制；部分类型可触发打开外部应用、在终端执行或浏览器执行脚本等动作，并在有风险的操作前提示确认。

### 4.5 JSON 备份与导入

- **导出 JSON**：将当前内存中的完整配置序列化为带版本与时间信息的文件，便于备份与分享。  
- **导入 JSON**：解析文件（支持「供应商数组」或带 `providers` 字段的对象两种外层结构），经标准化后**替换**当前全部配置并持久化。  

合并策略、冲突解决若与历史文档或 README 表述不一致，以当前实现为准；若产品后续增加「合并导入」等能力，应在实现变更后同步更新本文档。

---

## 5. 数据存储与安全边界

配置持久化在 Electron 用户数据目录下的本地文件（如 `configs.json`），应用主要在本地使用所填密钥发起检测与生成导出内容，不向业务方云端上传配置。涉及在终端执行或打开外部协议的导出，由用户在确认风险提示后自行承担环境风险。

---

## 6. 错误与反馈

导入格式错误、网络与检测失败、外部调起失败等情况，通过界面提示或模型行上的状态与详情呈现，便于用户排查；具体文案与交互不在本文展开。
