# 供应商弹窗与模型弹窗：自动填充说明

本文用纯文字说明「添加或编辑供应商」与「添加或编辑模型」两个界面里，**哪些内容会在什么情况下被自动填入**，以及**哪些内容不会被自动改掉**。为便于与配置、代码对照，**关键字段在文中会同时标注英文属性名**（与 `ProviderConfig`、`ModelConfig`、`ModelParams` 等一致）。

---

## 术语对照（中英字段名）

| 中文（界面/文中说法） | 英文（配置字段 / 类型） | 说明 |
|----------------------|-------------------------|------|
| 供应商（整条配置） | **provider** → `ProviderConfig` | 一条供应商记录 |
| 模型（单条配置） | **model** → `ModelConfig` | 隶属于某 provider 的 `models` 中的一项 |
| 供应商标识 | **`id`** | 供应商唯一标识，如 `openai` |
| 供应商展示名称 | **`name`** | 仅展示用 |
| 接口类型 | **`apiType`** | `openai` \| `anthropic` |
| Base URL（服务根地址） | **`baseUrl`** | 仅根部分，末尾通常不含路径段 |
| Endpoint（请求路径） | **`endpoint`** | 相对路径或路径段，按服务端实际填写 |
| URL（完整请求地址） | **Base URL + Endpoint** | 实际发起请求时的完整形态；配置里拆成 `baseUrl` 与 `endpoint` 两段 |
| 官网链接 | **`website`** | 可选 |
| 密钥 / API Key | **`apiKey`** | 可选 |
| 模型列表 | **`models`** | `ModelConfig[]` |
| 模型标识 | **`id`**（在模型上下文中为 model id） | 如 `gpt-5.4` |
| 模型展示名称 | **`name`** | 可选 |
| 模型能力与参数整块 | **`params`** → `ModelParams` | 可选扩展字段 |
| 上下文长度 | **`contextWindow`** | 对应常见说法 *context window* |
| 单次输出上限 | **`maxTokens`** | 与 UI 中 Max 输出等对应 |
| 是否推理 | **`reasoningMode`** | 布尔或空 |
| 输入类型 | **`inputTypes`** | 字符串数组 |

**命名约定**：文中与界面上的 **Base URL** 对应配置字段 **`baseUrl`**；**URL** 一词专指 **Base URL 与 Endpoint 拼在一起后的完整请求地址**（除非另行说明）。

---

## 一、预设数据从哪来、用在哪

应用里有一份**供应商预设**（**provider preset**）：每个供应商有一条记录，包含展示名称，以及**按接口类型（`apiType`）分别给出的**默认 Base URL（`baseUrl`）、Endpoint（`endpoint`）和接口类型本身。

另有一份**模型预设**（**model preset**）：每条记录对应一个模型标识（`id`）、展示名称（`name`）、所属供应商类别，以及可选的上下文长度（`contextWindow`）、单次输出上限（`maxTokens`）、是否推理（`reasoningMode`）、输入类型（`inputTypes`）等，映射到保存时的 `ModelParams`。

这两类预设随应用提供，在需要时被读入内存。

除此之外，**当前用户已经保存过的供应商和模型**也会参与模型标识下拉的组合：凡是在本地配置里出现过的模型 `id`，只要还没有被预设列表覆盖到，也会出现在下拉里，方便复用。

供应商弹窗里 **Base URL** 与 **Endpoint**（`baseUrl` / `endpoint`）两个带下拉的输入框，其**可选列表**来自：内置预设里汇总出的常见 Base URL 与 Endpoint，再加上各供应商预设里已经写过的 Base URL 与 Endpoint，合并去重后的结果。

## 二、供应商弹窗

### 打开时

- **新增供应商**：各字段为默认空态或默认值，**不会**根据空的供应商标识（`id`）去猜一整张表。
- **编辑已有供应商**：界面上的内容与当前已保存的这条供应商一致，相当于把已有配置原样拿来改。

### 供应商标识（`id`，可输入、也可从列表选）

当用户**输入结束并离开输入框**（失焦）时，若标识非空：

- 若在供应商预设里**找不到**这条标识：只自动填**供应商展示名称**（`name`），规则是把用户输入的、用连字符分隔的标识转成类似标题的读法（例如把分段首字母大写、用空格分开）。
- 若**能找到**对应预设：会写入预设里的 `name`；并尝试根据预设里「按接口类型区分的默认连接信息」填**接口类型、Base URL、Endpoint**（`apiType`、`baseUrl`、`endpoint`）。
  - 若用户**在失焦前已经选好了接口类型**（`apiType`），则优先选用与该类型一致的那一组默认 Base URL 与 Endpoint；若没有单独为这种类型配置，则用预设里的第一组。
- 若预设存在但其中没有任何一组连接信息：只自动填展示名称（`name`）。

当用户**从下拉列表里点选某一条预设供应商**时，逻辑与失焦类似：同样会尝试写入 `name`、`apiType`、`baseUrl`、`endpoint`。此时「优先匹配哪一种接口类型」以**点选瞬间界面上已选择的 `apiType`**为准；匹配规则同样是先对得上则用那一组，否则用预设中的第一组。

**官网链接**（`website`）和**密钥**（`apiKey`）不会因为上述操作被自动改掉，除非用户自己编辑。

### 接口类型（`apiType`，两种接口二选一）

用户在两种接口类型之间切换时：

- **`apiType`** 会立即变为用户新选的那一种。
- 若当前供应商标识（`id`）在预设中存在，且该预设里**为这种接口类型单独配过** `baseUrl` 和 `endpoint`，则这两项会换成预设里那一套。
- 若预设里**没有**为这种接口类型单独配置，则**只改 `apiType`**，用户已经填好的 `baseUrl` 和 `endpoint`**保持不动**。

### Base URL 与 Endpoint（`baseUrl` / `endpoint`）

这两项没有单独的「再算一遍」逻辑。它们要么来自上面**供应商标识或接口类型**触发的自动写入，要么来自用户手输，要么来自合并后的下拉补全列表。二者拼起来才是完整 **URL**（见术语表）。

## 三、模型弹窗

### 打开时

- **新增模型**：模型标识（`id`）和能力参数（`params`）为默认空态，不预先猜模型。
- **编辑已有模型**：显示该条已保存的模型信息。

### 模型标识下拉列表是怎么排的

首先，系统会在供应商预设里找**与当前这条供应商**「标识相同」或「展示名称一致（不区分大小写）」的那一条。

- 若**能找到**：把所有**属于同一供应商类别**的模型预设排在列表**前面**，其余模型预设排在**后面**。
- 若**找不到**：所有模型预设按原有顺序全部参与排序。

在此基础上，下拉里出现的模型标识包括：**上述排序后的预设里的模型 `id`**（去重并保持顺序），再加上**当前所有供应商里已经保存过、但还没出现在预设标识集合里的**本地模型 `id`。这样用户以前用过的模型标识也能被选到。

### 模型标识何时触发自动填充

**失焦**：在「当前供应商下、按上面规则排好序的整份模型预设列表」里，用去掉首尾空白后的 `id` 去对；对上第一条就用来补全。

**从下拉选中一项**：只在**当前选中的这一条**上匹配（若该项没有对应预设，则只更新 `id` 本身，不补其它字段）。

### 自动填充会改哪些、不改哪些

若根据模型 `id` 在当次使用的列表里找到了对应预设：

- **展示名称**（`name`）：仅当用户**还没填**展示名称时，才用预设里的名称补上；用户已经填过则不动。
- **能力与参数**（`params`：如 `contextWindow`、`maxTokens`、`reasoningMode`、`inputTypes` 等）：在**保留用户已填内容**的前提下，用预设里**有值的项**去补全或更新；预设里某一项没有有效值时，不强行覆盖用户已有内容。

若**找不到**对应预设，则不对其它字段做自动修改。

数值类一般在预设里为合理正数时才会写入；推理类（`reasoningMode`）需预设里明确是或否；输入类型（`inputTypes`）需预设里给出非空列表才会写入。

## 四、保存时

保存供应商或保存模型时，只做必填项检查和把当前表单内容写入配置，**不会**在保存这一刻再去合并一遍预设。自动填充只发生在用户在弹窗里输入、选择、切换的过程中。

## 五、边緣条件（实现备忘）

以下条目来自已实现行为与 Spec 对齐时的整理，**后续改代码时请勿漏掉**，否则容易出现「清空用户已填」或「错套其它接口类型」的回归。

### 5.1 何谓「一组有效的连接信息」

对预设里 `apiConfigs` 的每一条（给定 `apiType`、`baseUrl`、`endpoint`）：

- **有效**：`baseUrl` 与 `endpoint` 中**至少有一端**在去掉首尾空白后非空（允许只填 Base URL、或只填 Endpoint，视产品约定）。
- **无效**：两端都为空或仅空白 —— 该条**不**视为「已配置连接信息」，不得用来覆盖表单里的 Base URL / Endpoint。

判断时应对字符串做 `trim`，避免 JSON 里 `""` 或 `"  "` 误判为「有值」。

### 5.2 供应商标识失焦 / 从列表点选预设供应商

- 若整份预设里 **没有任何一条** `apiConfig` 具备 **5.1** 意义上的有效连接信息：只能自动填展示名 **`name`**（以及无预设时的「从 id 推导名称」），**不得**写入 `apiType`、`baseUrl`、`endpoint`，尤其**不得**把空字符串写入后两项，以免**清空用户已填**。
- 若当前界面已选的 **`apiType` 在预设里能命中一行**，但该行连接信息无效，而**其它** `apiType` 的行上有有效连接：**不得**为了「补全」而自动改用其它类型的那一行。否则会把 **`apiType` 与用户选择不一致**，或与「优先当前所选类型」的叙述冲突。此种情况应退化为**只保证 `name`**（与上一条「全局无有效连接」时行为一致，仅填名称相关字段）。

### 5.3 切换接口类型（`apiType`）

- 仅当预设里**存在**与**新选中的** `apiType` 对应的那一行，且该行在 **5.1** 意义上**有效**时，才用预设覆盖 **`baseUrl`、`endpoint`**。
- 若不存在该类型的配置行，或存在但该行无效：界面上的 **`apiType` 仍随用户切换**，但 **`baseUrl`、`endpoint` 保持用户当前已填内容**，不得用空预设行覆盖。

### 5.4 与上文条目的对应关系

- **第二节** 中「没有任何一组连接信息：只自动填展示名称」—— 按 **5.1** 判定「有没有一组」；满足「没有任何一组有效连接」时，行为同 **5.2** 第一段。
- **第二节** 中「没有为这种接口类型单独配置则只改接口类型」—— 含「有该行但连接无效」的情形，行为同 **5.3**。
