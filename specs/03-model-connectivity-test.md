# 连通性测试（Connectivity test）

**文档性质**：定义用户可感知行为与验收口径，不展开工程实现。

---

## 1. 目标与边界

- **目标**：验证“当前 Provider 的连接配置 + 当前 Model ID”在当前时刻是否可调用。
- **边界**：该能力不评估计费、配额、吞吐、长上下文极限、工具调用能力等，不替代服务方 SLA。

---

## 2. 测试输入与判定

- 输入来自模型所在 Provider 的连接信息与该模型的 Model ID。
- 系统发起一次“低成本最小探测”。
- 判定规则：
  - 服务返回成功时，判定为 **success**。
  - 服务返回失败或网络异常时，判定为 **error**，并记录可读错误摘要。

---

## 3. 与 API type 的关系

- 测试请求的鉴权方式由 **API type** 决定。
- 若用户把连接配置指向“兼容网关”，测试结果按该网关的兼容能力解释。
- 若目标是原生服务且与当前探测格式不兼容，结果可能失败；这是配置兼容性问题，产品需给出可读提示。

---

## 4. 结果展示与状态回写

- 每条 Model 至少展示：
  - 最近测试状态：**pending / success / error**
  - 最近测试时间
  - 最近测试信息（成功摘要或失败摘要）
- 若测试过程中该行模型已被用户改成另一个 Model ID，本次返回结果必须丢弃，避免写错行。
- 用户保存模型配置后，该模型测试状态重置为 **pending**，历史测试时间与信息清空。

---

## 5. 触发方式

- **单条测试**：在模型行点击测试按钮触发。
- **批量测试**：对当前 Selection 中所有模型触发。

批量测试的体验要求：

- 批量开始前，自动展开含选中模型的 Provider，便于用户观察结果。
- 同一模型在“正在测试”期间再次触发应被忽略，避免重复请求与状态抖动。

---

## 6. 失败处理

- 网络失败、超时、服务侧拒绝、格式错误等都属于失败路径。
- 失败必须可见：用户能在模型行看到错误状态与可读错误信息。
- 单条失败不应中断其它模型的测试结果回写。

---

## 7. 与远端模型列表的分工

- **远端模型列表**：负责“发现候选 Model ID”。
- **连通性测试**：负责“验证候选在当前配置下是否可用”。

二者配合形成“发现 -> 验证 -> 导出”的闭环。
