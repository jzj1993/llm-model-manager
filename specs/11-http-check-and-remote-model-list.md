# 网络交互的产品语义（Connectivity check & Remote model list）

目标：定义网络能力在产品层的“语义与验收口径”，不绑定具体接口路径、请求字段名或实现常量。

---

## 1. 完整请求落点（Service URL）

- Service URL 由 Base URL 与 Endpoint 组合得到。
- 组合规则必须对用户输入友好：
  - 自动处理斜杠衔接，避免出现双斜杠
  - 两侧都不带斜杠时自动补一个
  - 任一侧为空时以另一侧为准

---

## 2. 鉴权语义（与 API type 相关）

- 当 Provider 填写了 API Key：
  - OpenAI-compatible：使用该生态常见的密钥鉴权方式。
  - Anthropic-compatible：使用该生态常见的密钥鉴权方式，并满足其版本约束（若目标服务要求）。
- 当 Provider 未填写 API Key：
  - 仍允许发起请求（因为部分服务可能无需鉴权或通过其它方式鉴权），但失败风险由用户承担，错误信息需可见。

---

## 3. 连通性测试（Connectivity test）

### 3.1 目的

验证“以当前 Provider 的连接信息为准，Model ID 是否可被成功调用”。

### 3.2 探测策略（产品口径）

- 发送一次“最小探测请求”，满足：
  - 请求足够小（低成本、低延迟）
  - 能覆盖鉴权、路由到模型、以及服务端基本可用性
- 判定规则：
  - 若服务端返回成功（2xx）则标记 success
  - 否则标记 error，并展示可读错误摘要

### 3.3 兼容性声明

产品允许用户把任意兼容网关配置为 Endpoint。若某生态的原生 API 与当前探测形状不兼容，测试结果以“该 Endpoint 是否兼容本产品探测形状”为准。

---

## 4. 远端模型列表（Remote model list）

### 4.1 目的

从 Provider 的服务端“枚举可能可用的模型”，减少手工查找与录入 Model ID 的成本。

### 4.2 尝试策略（产品口径）

- 产品会按“常见约定”尝试少量候选的模型列表入口（例如带版本前缀与不带版本前缀两种）。
- 任一候选返回成功且可解析出列表，即视为成功；否则视为失败，可重试。

### 4.3 解析与映射

对每个返回项：

- **Model ID**：取服务端返回的主标识字段；若缺失则丢弃该项。
- **Model display name**：优先取服务端提供的展示名字段；缺失则回退为 Model ID。
- **Model params（可选）**：
  - 若服务端提供 context window / max tokens 等能力字段则带入
  - 若服务端未提供，则可按“模型名称启发式”补充默认值（见 4.4）
- 初始测试状态为 pending（未测试）

### 4.4 名称启发式（Heuristics）

- Reasoning mode：若模型名包含常见推理关键词（例如 reasoning、thinking、o 系列、r1 等），可推断为启用；否则保持未定义。
- Input types：若模型名包含常见多模态关键词（例如 vision、omni、image 等），可推断支持 text+image；否则默认仅 text。

---

## 5. 与 UI 的衔接

- 远端模型列表用于“发现”，连通性测试用于“验证”。
- 远端列表弹窗中应标记“已添加”的模型（按 Model ID 判等），并禁止再次勾选。
- 用户确认添加后，应将选中项追加到该 Provider 的 Models 末尾并持久化；全局 Selection 需清空以避免误操作。
