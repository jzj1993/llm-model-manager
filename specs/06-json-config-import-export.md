# JSON 备份与导入

**文档性质**：定义用户可见的备份/恢复策略与数据安全边界。

---

## 1. 备份（导出 JSON）

- 备份文件应包含：
  - 版本标识
  - 导出时间
  - 完整 Provider 集合
- 交付形态为下载文件，文件名建议带时间信息，便于区分多次备份。
- 用途：换机迁移、回滚、协作传递、人工审阅后再导入。

---

## 2. 恢复（导入 JSON）

导入策略采用 **整库替换**：

- 解析成功并通过校验后，用导入内容替换当前全部配置。
- 替换后清空 Selection，避免保留旧工作集的选择状态。

接受两种外层结构：

- 顶层直接是 Provider 列表
- 顶层是对象，且对象内包含 Provider 列表字段

---

## 3. 健壮性与防误伤

- 文件解析失败时必须提示用户，不能静默失败。
- 缺少有效 Provider 列表时必须拒绝导入并提示，禁止把当前配置静默覆盖为空。
- 导入流程中任意失败都不应修改当前已保存数据。

---

## 4. 备份文件与本地运行时存储的区别

- 备份文件用于“跨环境迁移与版本识别”，包含版本与导出时间等元信息。
- 本地运行时存储用于“应用日常读写”，只需保存当前工作集本体。

两者用途不同，格式可不同，但语义必须一致。

---

## 5. 敏感信息与分享风险

- 备份文件可能包含 API Key 等敏感字段。
- 分享前应提醒用户确认脱敏需求与传播范围。

---

## 6. 对外说明一致性

- 对外文档必须明确“导入是整库替换”。
- 若后续新增“合并导入”等模式，必须在 UI、规格与 README 同步说明差异与适用场景。
